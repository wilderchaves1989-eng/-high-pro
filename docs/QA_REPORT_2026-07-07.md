## CRITICO

### LOGIN-01 | CRITICO | Login nao da acesso — utilizadores nao existem/nao confirmados
Confirmado via /auth/v1/settings do projeto:
- disable_signup: false (registo aberto)
- mailer_autoconfirm: false (exige confirmacao de email)

O frontend usa supabase.auth.signInWithPassword. Para logar, o utilizador tem de
existir no Supabase Auth E ter o email confirmado. Nada no repositorio cria esses
utilizadores no Supabase Auth:
- O trigger handle_new_user so cria a linha em `profiles` DEPOIS que um utilizador
  Auth e criado — nao cria o utilizador.
- O backend/prisma/seed.js pertence ao backend Node antigo (tabela `users` com
  bcrypt), sem relacao com o Supabase Auth usado em producao. As senhas ali
  (admin/atd2025/prof2025) NAO valem no site.
- As credenciais em Desktop/HIGH PRO GESTAO ESCOLA/credenciais.txt
  (admin@highpro.pt / admin123456 etc.) provavelmente nunca foram criadas no
  Supabase Auth; e mesmo que tenham sido via "Criar Acesso", o email @highpro.pt
  nao recebe a confirmacao -> conta nunca confirmada -> login falha.

CORRECAO (imediata):
1. Supabase Dashboard -> Authentication -> Users -> Add user.
2. Preencher email + password e MARCAR "Auto Confirm User".
3. Usar um email real (ex.: atendimento.highproformacoes@gmail.com) para o admin.
4. Confirmar que existe a linha correspondente em `profiles` com perfil='GESTOR'
   (o trigger cria; se nao criar, inserir manualmente).

### SEC-01 | CRITICO | Autorizacao aberta — qualquer pessoa vira GESTOR com acesso total
Duas falhas combinadas:
(a) Registo aberto (disable_signup=false) + a chave anon esta publica no bundle JS.
(b) O trigger confia no metadata do cliente:
    COALESCE(NEW.raw_user_meta_data->>'perfil', 'ATENDIMENTO')
Assim, qualquer estranho pode chamar /auth/v1/signup com
user_metadata.perfil="GESTOR", confirmar o proprio email, e o trigger cria um
profile GESTOR. Como as policies RLS sao "FOR ALL TO authenticated USING (true)",
esse utilizador ganha leitura/escrita/exclusao de TODAS as tabelas (alunos, aulas,
cursos, config, profiles).

Alem disso, mesmo sem signup: a policy de UPDATE em profiles e USING(true) sem
WITH CHECK, entao um utilizador ATENDIMENTO consegue dar UPDATE no proprio profile
e mudar perfil para GESTOR. Os checks de papel (isGestor) existem so no frontend e
sao contornaveis chamando o PostgREST direto.

CORRECAO:
1. Desligar signups publicos (Authentication -> Providers -> Email -> Disable
   "Allow new users to sign up"), OU restringir por dominio.
2. NUNCA confiar em raw_user_meta_data para perfil no trigger; forcar
   perfil='ATENDIMENTO' (ou o menor privilegio) e promover so via GESTOR.
3. Reescrever as policies RLS por papel: por exemplo, so GESTOR pode INSERT/UPDATE
   /DELETE em profiles, config e cursos; impedir que um user altere o proprio perfil
   (WITH CHECK que bloqueia mudanca de perfil por nao-GESTOR).

### SEC-02 | CRITICO | "Criar Acesso" (Config) esta quebrado e desloga o admin
api.js users.criar chama supabase.auth.admin.createUser. A admin API exige a chave
service_role, que NAO existe no frontend (so a anon). Logo supabase.auth.admin e
undefined -> optional chaining -> cai no fallback supabase.auth.signUp. Efeitos:
- signUp cria uma NOVA sessao e substitui a sessao atual: o admin e "deslogado" e
  passa a estar logado como o utilizador recem-criado.
- Com mailer_autoconfirm=false, o novo utilizador nem consegue logar ate confirmar
  email.
Resultado: a tela de gestao de credenciais nao cria acessos utilizaveis e sequestra
a sessao de quem a usa.

CORRECAO: criar utilizadores via Edge Function (ou backend) usando service_role,
ou orientar a criacao pelo Dashboard.

## ALTO

### SEC-03 | ALTO | Injecao de filtro PostgREST na busca de alunos
api.js alunos.listar:
  query.or(`nome.ilike.%${busca}%,email.ilike.%${busca}%,telefone.ilike.%${busca}%`)
`busca` (input livre do utilizador) e concatenado cru na string de filtro .or().
Virgulas/parenteses no texto podem quebrar a query ou injetar condicoes extras de
filtro (nao e SQL injection — o PostgREST protege o SQL — mas e injecao no filtro).
CORRECAO: escapar/validar `busca` (remover virgulas e parenteses) ou usar filtros
parametrizados.

### BUG-01 | ALTO | Mensagens de erro nunca aparecem (residuo de axios)
Varios catch usam err.response?.data?.error (ConfigPage.salvarCurso/salvarCred,
AlunosPage.salvar). A API agora e supabase-js, cujo erro e { message, code, ... }
sem `.response`. Logo qualquer erro real cai no fallback generico 'Erro' e a causa
real (ex.: email duplicado, RLS negado) fica escondida.
CORRECAO: usar err.message (ou err.details) nos alerts.

### BUG-02 | ALTO | Exclusao de aluno/curso sem verificacao de papel no cliente
AlunosPage esconde o botao Excluir para nao-editores, mas cursos/config nao checam
papel, e — mais importante — a verdadeira barreira deveria ser o RLS (ver SEC-01),
que hoje permite tudo a qualquer autenticado. Sem RLS por papel, esconder botoes
nao protege os dados.
CORRECAO: aplicar autorizacao no RLS (Postgres), nao so na UI.

## MEDIO

### DATA-01 | MEDIO | Sem paginacao em alunos.listar
Carrega todos os alunos de uma vez. Aceitavel com pouco volume; degrada com
milhares de registos. CORRECAO: .range()/paginacao server-side.

### UX-01 | MEDIO | Sem estados de carregamento/erro nas paginas
As paginas fazem .catch(() => {}) silencioso; falhas nao dao feedback ao utilizador
(tela vazia sem explicacao). CORRECAO: estados de loading e mensagens de erro.

### ORG-01 | MEDIO | Artefactos legados confusos no repositorio/Desktop
Coexistem: backend Prisma antigo (backend/), server.js/monolito antigo, e o
credenciais.txt com credenciais que nao valem em producao. Isso induz a erro (foi a
origem da confusao do login). CORRECAO: remover/arquivar o que nao pertence a stack
Supabase e atualizar credenciais.txt com a realidade.
